# DocuMind Operating Runbook

The operator's reference for running DocuMind: install, day-2 operations,
backup/restore, secret rotation, TLS troubleshooting, limitations, and upgrades.

DocuMind runs as four Docker Compose services — `caddy` (the only one with host
ports), `api` (FastAPI + the in-process ingest worker), `web` (Next.js), and
`postgres` (relational **and** vectors via pgvector). All configuration lives in
a single `.env` at the repo root, generated by `install.sh` and **gitignored**.

Commands below assume you are in the install directory (default `/opt/documind`).
A convenience wrapper is provided as `make` targets; the raw form is:

```bash
docker compose --env-file .env -f deploy/docker-compose.yml <cmd>
```

---

## 1. Install / first run

```bash
curl -fsSL https://raw.githubusercontent.com/Alighaemi9731/documind/main/install.sh | bash
```

You are prompted for a **domain** (apex A/AAAA record pointing at the box), an
**admin email** (login + Let's Encrypt account), and an optional **Gemini key**.
The installer then, idempotently:

1. preflights Docker, the GHCR images, port 80, DNS, and (on ≤2 GB RAM) creates a
   **2 GB swapfile**;
2. generates and **preserves** secrets (`POSTGRES_PASSWORD`, `JWT_SECRET`,
   `MASTER_KEY_FERNET`) — a re-run never regenerates them, so sessions and
   certificates survive;
3. `docker compose pull` → `up -d` → waits for Postgres → `alembic upgrade head`
   → seeds the operator Gemini key (if given) → ensures the bootstrap admin →
   waits for `/api/health/ready`;
4. probes `https://<domain>/api/health/ready` and prints the URL, or surfaces the
   Caddy/ACME logs if the certificate hasn't issued yet.

**Unattended:** set `DOMAIN`, `ADMIN_EMAIL`, (optional) `GEMINI_KEY` in the
environment and the installer won't prompt. Useful overrides: `DOCUMIND_DIR`,
`IMAGE_TAG`, `DOCUMIND_ACME_STAGING=1` (staging CA for smoke tests),
`DOCUMIND_SKIP_SWAP=1`.

After install, sign in at `https://<domain>` with the admin email (open
registration by default — see the admin dashboard to change the mode).

## 2. Day-2 operations

| Task | Command |
| --- | --- |
| Status | `docker compose --env-file .env -f deploy/docker-compose.yml ps` |
| Logs (follow) | `… logs -f api` (or `caddy`, `web`, `postgres`) |
| Restart a service | `… restart api` |
| Apply DB migrations | `make migrate` |
| Re-seed/rotate operator key | `… exec api python -m app.cli seed-operator-key --key <KEY>` |
| Create/ensure an admin | `… exec api python -m app.cli bootstrap-admin --email you@example.com` |
| Upgrade | re-run `install.sh` (pulls new `IMAGE_TAG`, preserves secrets/data) |

Admin tasks that have a UI (approvals, invites, quotas, usage, branding, operator
key) live in the **admin dashboard** at `/admin` — prefer it over the CLI.

## 3. Backup & restore

Back up three things **together** so a restore is consistent: the Postgres
database, the uploaded-files volume, and `caddy_data` (the ACME account + issued
certificates, so HTTPS survives a restore without re-issuing).

```bash
# Backup -> ./backups/db-<stamp>.dump.gz + volumes-<stamp>.tar.gz (keeps last 7)
make backup            # = deploy/backup/backup.sh
# off-box copy (do this!):  rsync -a backups/ you@store:/documind-backups/
```

**Restore** onto a fresh install (Postgres must be up; `api` and `caddy` are
stopped during the restore and restarted after):

```bash
deploy/backup/restore.sh backups/db-<stamp>.dump.gz backups/volumes-<stamp>.tar.gz
```

Notes:
- `restore.sh` is destructive (`pg_restore --clean --if-exists`, and it **clears**
  the uploads + `caddy_data` volumes before extracting). It validates both input
  files, then requires you to type **`yes`**; set `RESTORE_YES=1` to skip the
  prompt in automation.
- Volume names are project-scoped (`documind_uploads`, `documind_caddy_data`);
  the scripts already use those names.
- Schedule `make backup` from cron (e.g. nightly) and verify a restore on a
  throwaway host periodically — an untested backup is not a backup.

## 4. Rotate secrets

### `MASTER_KEY_FERNET` (BYOK + operator-key encryption at rest)
Rotation is **MultiFernet**: the variable holds a comma-free single key normally;
to rotate, list the **new key first**, then the old key(s), re-encrypt, then drop
the old key.

```bash
# 1. Generate a new key (url-safe base64, 32 bytes):
NEW=$(openssl rand -base64 32 | tr '+/' '-_')
# 2. Put NEW first, keep the OLD for decryption (comma-separated):
#    MASTER_KEY_FERNET=$NEW,$OLD       (edit .env)
# 3. Re-encrypt every at-rest secret under the new key:
docker compose --env-file .env -f deploy/docker-compose.yml \
  exec api python -m app.cli rotate-master-key
# 4. Drop the OLD key:  MASTER_KEY_FERNET=$NEW   (edit .env) and restart api.
```
The app accepts a comma-separated `MASTER_KEY_FERNET` (decrypt with any, encrypt
with the first). Skipping step 3 and removing the old key will make existing
encrypted keys undecryptable — always rotate before retiring a key.

### `JWT_SECRET`
Changing `JWT_SECRET` invalidates all outstanding access **and** refresh tokens —
every user must log in again. Edit `.env`, then `restart api`. Do this only on
suspected compromise; there is no graceful overlap.

### `POSTGRES_PASSWORD`
Rotating the DB password requires changing it in Postgres **and** `.env` together
(`ALTER ROLE documind WITH PASSWORD …` then update `.env`, `restart`). Prefer a
backup→restore onto a fresh install if you're unsure.

## 5. ACME / HTTPS failure runbook

Symptom: `https://<domain>` doesn't load, or the installer's final probe warned.
Caddy obtains certificates via the Let's Encrypt **HTTP-01** challenge on port 80.

```bash
docker compose --env-file .env -f deploy/docker-compose.yml logs --tail 50 caddy
```

| Cause | Check | Fix |
| --- | --- | --- |
| DNS not pointing here | `dig +short A <domain>` vs this host's public IP | Fix the A/AAAA record; wait for TTL, then `restart caddy` |
| Port 80/443 blocked | `curl -sI http://<domain>/` from elsewhere; cloud firewall/security group | Open inbound 80 **and** 443 (TCP; 443 UDP too for HTTP/3) |
| LE rate limit hit | caddy logs show `too many certificates`/`rateLimited` | Wait out the window, or test with `DOCUMIND_ACME_STAGING=1` re-run, then switch back |
| Wrong/stale account email | `ACME_EMAIL` in `.env` | Correct it; `restart caddy` |

For debugging without burning the production rate limit, re-run the installer
with `DOCUMIND_ACME_STAGING=1` (staging certs are **untrusted** by browsers — use
only to confirm issuance, then re-run without it to get a real certificate).
`caddy_data` persists the account + certs across restarts, so a healthy install
won't re-hit ACME on every boot.

## 6. Known limitations (v1)

- **No OCR.** Image-only / scanned PDFs with no embedded text layer fail fast
  with `NO_TEXT` — they are not transcribed. Provide text-based PDFs/DOCX/TXT/MD.
- **No SMTP.** Invites are **copyable links** the operator relays; self-service
  approval is in-app (a "pending" state + an admin pending-count badge). ADR-0016.
- **Single-turn retrieval.** Chat history is persisted, but each question is
  retrieved independently (no history-aware query rewriting). ADR-0017.
- **Local embeddings need ≥4 GB.** `bge-m3` (`ENABLE_LOCAL_EMBEDDINGS=true`) is
  opt-in and refused under 4 GB; the default is the Gemini embedding API (~0
  local RAM).
- **Single host.** The default profile is one box; there is no built-in HA /
  multi-node clustering.

## 7. Upgrading & rollback

Upgrades are just a re-run: bump `IMAGE_TAG` (or keep `latest`) and run
`install.sh` again. It pulls the new images, re-applies migrations, and preserves
your secrets and data. To roll back, set `IMAGE_TAG` to the previous version and
re-run — but note migrations are **forward-only**; a rollback that crosses a
schema change must be paired with a database restore from before the upgrade
(§3). Always `make backup` before a major upgrade.

## 8. Resource notes (2 GB VPS)

Postgres is tuned small (`shared_buffers=256MB`, `work_mem=8MB`,
`maintenance_work_mem=64MB`, `max_connections=20`, `jit=off`) and every service
has a `mem_limit`. The mandatory swapfile absorbs the ingest + HNSW-index-build
peak so Postgres is never OOM-killed. If you see OOMs, confirm swap is active
(`swapon --show`) and that `INGEST_CONCURRENCY=1`. The HNSW index builds lazily
on first ingest — the first upload into a new project is slower than subsequent
ones.
